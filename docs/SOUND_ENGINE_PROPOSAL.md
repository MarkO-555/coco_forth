# Non-Blocking Sound Engine — Proposal

*May 2026 — Paul Cunningham + Claude*

## Why this exists

Today's `forth/lib/sound.fs` is **synchronous**: every word in the
library (`snd-tone`, `snd-saw`, `snd-sin`, `snd-noise`, the
convenience effects) blocks the caller until the sound finishes.
That works fine for short interjections (a UI beep, a hit-tick, a
boom on player death) where pausing the game for 30–300 ms is
imperceptible or even desirable.

It does not work for a real game.  In a typical game frame the CoCo
has ~14,800 cycles to:

- read input and update game state
- redraw the playfield (often the dominant cost)
- handle physics, AI, collision
- composite sprites
- update score / HUD

The frame budget is already tight before sound enters the picture.
A blocking 950 ms `snd-tone` call is fine for the main menu but
unusable inside an action loop.  We need a sound engine that fits
into the frame's spare cycles **without** stopping the game.

## The DAC, in numbers

CoCo audio is generated by writing the 6-bit DAC at `$FF20`.  How
fast you have to update it determines what you can produce.

```
CPU clock:        0.894 MHz   (1 cycle ≈ 1.12 µs)
NTSC frame:       16.67 ms    = 14,800 cycles
HSYNC row:        63.5 µs     =     57 cycles  (240 rows/frame)
```

**Square waves** need the DAC toggled at twice the audio frequency:

| Audio freq | Toggles/sec | Period | Toggles/frame |
|-----------:|------------:|-------:|--------------:|
| 100 Hz     | 200         | 5.0 ms | 3             |
| 500 Hz     | 1,000       | 1.0 ms | 17            |
| 1 kHz      | 2,000       | 500 µs | 33            |
| 4 kHz      | 8,000       | 125 µs | 133           |

**Sample-based playback** (sawtooth, sine, noise) needs a new DAC
write at the sample rate:

| Sample rate | Quality                | Writes/frame |
|------------:|------------------------|-------------:|
| 4 kHz       | tinny but audible      | 67           |
| 8 kHz       | "telephone band"       | 133          |
| 16 kHz      | actual music territory | 267          |

A toggle is 1 store + housekeeping (~30 cycles); 17 evenly-spaced
toggles per frame consume ~510 cycles, ~3.5 % of the frame budget.
That is the right ballpark for an action-game sound engine.

## Architectural options

### A. Per-frame burst engine (recommended starting point)

The game calls a single word `snd-tick` once per video frame
(directly after the existing VSYNC wait).  `snd-tick` does N
evenly-spaced DAC writes within its own runtime, then returns.

- **Cost:** ~16 toggles × 30 cy = ~500 cy/frame ≈ **3.4 %** of frame
- **Top frequency:** ~500 Hz square wave (one full cycle every two
  frames is well above audible; the limit is intra-frame toggle
  spacing, ~1 ms)
- **Programming model:** a state byte holds the current pitch /
  duration; the game writes new state any time, `snd-tick`
  consumes it
- **Pros:** trivial to integrate, no IRQs, no special timing
- **Cons:** can't do sample playback (8 kHz+); no polyphony

### B. HSYNC-polled engine

Game polls `$FF01 bit 7` (HSYNC flag) between work units, and on
each row tick writes one DAC sample.

- **Cost:** ~30 cy per row × 240 rows = ~7,200 cy/frame ≈ **49 %**
  of frame, **but** spread out — only sampled when the game
  voluntarily yields
- **Top sample rate:** 15.7 kHz (full audio bandwidth)
- **Programming model:** the game must intersperse polls; structured
  awkwardly compared to a clean per-frame tick
- **Pros:** real audio quality without IRQs
- **Cons:** intrusive on game code; misses ticks if the game does
  long un-yielded work

### C. HSYNC IRQ-driven engine

Install a FIRQ handler on PIA0 CA1 (HSYNC).  On each IRQ, fetch the
next sample from a queue and store to `$FF20`.  Game thread refills
the queue every frame.

- **Cost:** ~30 cy per IRQ × 15,700 IRQs/sec ≈ **52 % CPU**
  (always-on, even if no game work pending)
- **Top sample rate:** 15.7 kHz, fully sampled audio
- **Programming model:** game just refills a circular buffer of
  samples once per frame
- **Pros:** real audio, multiple voices possible (sum samples
  before write), no synchronization burden on game code
- **Cons:** big CPU cost; existing kernel masks IRQs (`ORCC #$50`)
  and would need a careful unmask + handler-install path; conflicts
  with anything else that wants HSYNC

### D. VSYNC IRQ tick

VSYNC fires at 60 Hz (PIA0 CB1).  Useful as a *clock source* for
advancing music sequencer state, but 60 Hz is far below the audible
range — VSYNC alone cannot drive a tone.  This is a complement to A
or C, not a replacement.

## Recommendation

Ship **option A** as the next iteration.  It is the smallest
delta against today's library, the cheapest at runtime, and
sufficient for arcade-game beeps, lasers, footsteps, and a
simple monophonic music engine.  It also leaves option C as a
future upgrade path without architectural rework.

## API sketch

```forth
\ Engine state, all 1-byte fields exposed as kvar-* via fc.py
VARIABLE snd-state    \ 0=idle, 1=tone, 2=noise, 3=sweep
VARIABLE snd-pitch    \ current half-cycle delay (16-bit)
VARIABLE snd-frames   \ frames remaining (16-bit; 0 = done)
VARIABLE snd-toggle   \ next-toggle countdown (16-bit, in cycles)
VARIABLE snd-dac      \ current DAC value ($FC or $00)

\ Public API ─────────────────────────────────────────────────────────

\ Begin a tone playing in the background.  Returns immediately.
: snd-play  ( pitch frames -- )
  snd-frames !
  snd-pitch !
  snd-init                       \ PIA setup once
  1 snd-state ! ;

\ Per-frame tick.  Game calls this exactly once after VSYNC.
\ Cost ~500 cy when active, ~30 cy when idle.
CODE snd-tick
        ;; If snd-state = 0, return immediately.
        LDA     snd-state
        BEQ     SNT_OUT
        ;; Otherwise: do N evenly-spaced toggles within this frame,
        ;; using snd-pitch to set the inter-toggle interval.
        ;; ... (16-iteration toggle loop with phase accumulator) ...
        ;; Decrement snd-frames; if zero, set snd-state = 0.
SNT_OUT ;NEXT
;CODE

\ Convenience wrappers (compose snd-play with reasonable defaults).
: snd-beep!   ( -- )  300 8 snd-play ;
: snd-laser!  ( -- )  ... ;     \ schedules a sweep
```

The trailing `!` in the convenience names signals "fire-and-forget,
returns immediately" to distinguish from the synchronous `snd-beep`
already in the lib.

## Cost analysis

Frame breakdown for a representative action game with the engine
active:

| Component        | Cycles | Frame % |
|------------------|-------:|--------:|
| VSYNC wait       |    ~50 |    0.3% |
| `snd-tick`       |   ~500 |    3.4% |
| Game logic       | (rest) |   96.3% |

The engine adds **~3.4 %** to every active frame and ~0.2 % to
silent frames (`snd-state = 0` early-out).  That is well under the
budget of any action game that has time to draw at all.

## Implementation plan

1. **Issue + branch.** File a single tracking issue ("Async sound
   engine: per-frame burst") that owns the work and references this
   document.
2. **Kernel-side prep.** No kernel changes needed for option A — all
   state lives in app-side `VARIABLE`s.  (Future option C work would
   want a kernel-side IRQ handler.)
3. **Library `forth/lib/sound-async.fs`.** New file — keeps the
   existing synchronous lib untouched while we iterate.  Ports the
   PIA setup, defines `snd-play`, `snd-tick`, plus a small set of
   fire-and-forget convenience words mirroring today's `snd-*`.
4. **Demo `src/sound-async/`.** A new demo that runs a moving
   sprite + ongoing tone simultaneously, proving the engine doesn't
   block the game loop.
5. **Migration optional.** Existing demos that use `snd-beep` etc.
   keep working; the async lib is purely additive until / unless we
   decide to replace the synchronous one.

## Open questions

- **Multi-channel?** Option A produces one square wave at a time.
  A second voice doubles the toggle cost (~1,000 cy/frame, 7 %)
  and requires interleaving — worth doing in a v2 once the v1 lands.
- **Volume control?** The 6-bit DAC supports 64 levels.  Today's
  library only ever writes `$FC` and `$00` — full volume.  An async
  engine could parameterise the "high" sample for crude volume.
- **Music engine on top.** Once `snd-play` exists, a sequencer
  (read note from a buffer, set pitch, set frame count, advance)
  is straightforward — that's what #37 (PLAY music command) would
  be built on.
