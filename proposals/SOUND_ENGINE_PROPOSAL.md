# Non-Blocking Sound Engine — Proposal

*May 2026 — Paul Cunningham + Claude*
*Revised 2026-05-16 after the design session that picked the HSYNC IRQ path.*
*Revised 2026-06-09: the HSYNC-FIRQ premise is a hardware error (HSYNC is on
IRQ, not FIRQ). See "Revision: HSYNC is IRQ — cooperative sub-frame scheduling"
below, which supersedes the Option C recommendation. Companion chart:
`proposals/hsync_polling_budget_chart.html`.*

## Why this exists

Today's `lib/sound.fs` is **synchronous**: every word in the
library (`snd-tone`, `snd-saw`, `snd-sin`, `snd-noise`, the
convenience effects) blocks the caller until the sound finishes.
That works fine for short interjections (a UI beep, a hit-tick, a
boom on player death) where pausing the game for 30–300 ms is
imperceptible or even desirable.

It does not work for a real game. Even a generous frame budget on
the CoCo is tight before sound enters the picture, and a blocking
950 ms `snd-tone` call is unusable inside an action loop. We need a
sound engine that fits into the frame's spare cycles **without**
stopping the game — async by construction.

## Frame budget — R0=1 partial-speed mode

This is the model the rest of the document assumes. A game like
Space Warp runs the kernel out of ROM and toggles SAM R0 (`$FFD7`)
to put the CPU into partial-speed mode: ROM accesses execute at
1.79 MHz, RAM accesses at 0.894 MHz. The blended effective rate
for a typical workload is **~1.34 MHz**, so each NTSC frame holds:

```
CPU clock:        0.894 MHz baseline, ~1.34 MHz effective with R0=1
NTSC frame:       16.67 ms = ~22,200 cycles  (was 14,800 at baseline)
HSYNC row:        63.5 µs = ~85 cycles      (was 57 at baseline)
HSYNC rate:       15.7 kHz (262 IRQs/frame, hardware-fixed)
VSYNC rate:       60 Hz
```

Space Warp's main loop currently consumes roughly **14,700 cycles**
per frame. That leaves **~7,500 cycles** for sound — and that's the
hard budget every option below has to live inside. The "1/3 of frame
for sound" ceiling is non-negotiable: anything more and the game
visibly slows down, which is why Space Warp shipped without sound at
all.

## The DAC, in numbers

CoCo audio is generated by writing the 6-bit DAC at `$FF20`. How
fast you update it determines what you can produce.

**Square waves** need the DAC toggled at twice the audio frequency:

| Audio freq | Toggles/sec | Period | Toggles/frame |
|-----------:|------------:|-------:|--------------:|
| 100 Hz     | 200         | 5.0 ms | 3             |
| 500 Hz     | 1,000       | 1.0 ms | 17            |
| 1 kHz      | 2,000       | 500 µs | 33            |
| 4 kHz      | 8,000       | 125 µs | 133           |

**Sample-based playback** needs a new DAC write at the sample rate:

| Sample rate | Quality                        | Writes/frame |
|------------:|--------------------------------|-------------:|
| 4 kHz       | tinny but musical              | 67           |
| 8 kHz       | "telephone band"               | 133          |
| 15.7 kHz    | full HSYNC rate, near-CD-ish   | 262          |

For bleep/bloop wavetable audio in a space shooter, a **4 kHz update
rate (~2 kHz audio bandwidth)** is plenty.

## Architectural options

### A. Per-frame VSYNC burst — *not viable for continuous tones*

The original proposal claimed `snd-tick` running once per VSYNC could
produce 500 Hz continuous square waves in ~500 cy/frame. That is
internally inconsistent: a 500 Hz tone needs 16+ evenly-spaced
toggles across the 16.7 ms frame, which means `snd-tick` either
blocks the main loop for the whole frame (~14,000 cy, not 500), or
bursts all 16 toggles in 500 cy at frame start — producing a 30 kHz
buzz packet followed by 16 ms of silence (audible 60 Hz envelope, not
a tone).

VSYNC burst mode therefore only produces **clicks, pops, and short
percussive elements**, not continuous wavetable tones. Useful but
fundamentally limited; not what this proposal targets.

### B. HSYNC polling

Main loop reads `$FF01` bit 7 between work units; on each transition
it writes a DAC sample. Period-era CoCo code (Glen Hewlett's tutorial
series, Daggorath's sound effects) leans on this because it dodges
FIRQ entry/exit cost entirely.

- **Cost when active:** ~15 cy per HSYNC sampled, but the main loop
  must yield to the poll often enough to catch every line
- **Top sample rate:** up to full 15.7 kHz if main loop yields after
  every short work block
- **Pros:** zero IRQ infrastructure, no register-save discipline
- **Cons:** intrusive — every loop body has to be structured around
  polling cadence. Misses ticks the moment the game does any work
  longer than a row period (~85 cy). Not actually async.

### C. HSYNC FIRQ — **recommended**

PIA0 CA1 is wired to the HSYNC signal. We install a FIRQ handler that
runs once per scan line, writes one byte to the DAC, acks the PIA,
and returns. FIRQ (not IRQ) is the right vector because it
auto-stacks only CC and PC — half the entry cost of IRQ.

- **Cost:** ~7,100 cy/frame when active (analysis below)
- **Top sample rate:** 15.7 kHz hardware-fixed; effective audio
  bandwidth determined by how often main loop updates the
  next-sample byte
- **Pros:** truly asynchronous — main loop is interrupted, not
  polled; sound stays in time-base regardless of game work
- **Cons:** non-zero overhead even when silent unless we
  enable/disable the CA1 IRQ around active notes (we do); kernel
  currently masks IRQs with `ORCC #$50`, so handler-install needs a
  careful unmask path

### D. VSYNC IRQ tick

VSYNC fires at 60 Hz (PIA0 CB1). Useful as a *clock source* for
advancing music-sequencer state (note → next note, envelope step),
but 60 Hz is far below audio rates — VSYNC alone cannot drive a
tone. A complement to C, not a replacement.

## Revision (2026-06-09): HSYNC is IRQ — cooperative sub-frame scheduling

The Option C recommendation rests on a hardware error. The proposal claims
"FIRQ (not IRQ) is the right vector" for HSYNC and the entire 7,074 cy/frame
cost model assumes FIRQ's cheap 10-cycle entry. **HSYNC does not fire FIRQ.**
The in-tree technical reference is unambiguous:

- PIA0 / U8 (`$FF00-$FF03`): CA1 = HSYNC, CB1 = VSYNC — both wired to the CPU
  **IRQ** line (`coco-guides/coco_technical_reference.txt:4775`,
  `coco-guides/vdg-modes.md:180-181`).
- PIA1 / U4 (`$FF20-$FF23`): CB1 = cartridge — the only stock **FIRQ** source
  (`coco_technical_reference.txt:4856`).

You cannot reroute HSYNC to FIRQ in software. On IRQ the CPU auto-stacks the
full 12-byte machine state, so the honest per-line cost is ~19 cy entry +
~10 cy body + ~15 cy RTI ≈ **~44 cy — even on a line where the handler decides
"not my turn" and bails.** At 262 lines that is **~11,500 cy/frame just to enter
and leave the handler**, before any sound work — ~52% of an R0-high frame, gone.

This kills the per-line interrupt approach and reframes the design:

- **"Output sound on only 1/3 of calls" saves work, not the tax.** PIA0 can't
  deliver every 3rd HSYNC; you take all 262 and skip-count in software, paying
  the ~44 cy entry on every skipped line too.
- **Interrupt off-slots are negative-value real estate.** Moving any existing
  main-loop task into an IRQ slot makes it *more* expensive — the main loop has
  no per-call entry tax. IRQ is only worth its cost for work that must *preempt*,
  and essentially nothing in a frame-locked CoCo game does.

### Cooperative cyclic executive — the viable path (revised recommendation)

What the per-line interrupt was reaching for — regular sub-frame cadence — is
better delivered **cooperatively**, with zero interrupt infrastructure. The
main loop's phase structure *is* the schedule; you place `snd-poll` calls at
phase boundaries. Three facts make this small:

1. **No kernel change.** Cooperative polling reads `$FF01` / writes `$FF20`
   directly from a library CODE word. No FIRQ vector, no `kvar-firq`, no
   `ORCC`/`ANDCC` unmask dance. `async-sound.fs` is self-contained.

2. **Don't target every HSYNC.** Bleep/bloop audio needs ~16-32 samples/frame
   (#485 / space-warp issue 485): 480-720 Hz square ceiling. That is a fresh
   DAC write every **~700-1,375 cy**, not every 84 cy. The constraint becomes
   *"no gap between consecutive `snd-poll`s exceeds ~1,375 cy."*

3. **Only four task types pierce that line.** Across the even/odd frame cycle:
   `collisions` (1,833 cy, odd), `jov-think` (1,449 cy, even), `bg-jov`
   (2,000 cy, both), `post-render` (1,443 cy, both) — three in any given frame.
   Each is < 2× the slot, so each needs exactly **one** split. Three split
   naturally at entity granularity (`bg-jov` → per-Jovian 3×~667 cy;
   `draw-jovians` is already a 3-Jovian `DO/LOOP`; `post-render` → between
   sub-tasks; `collisions` → by target pair). Only `jov-think` (sequential AI)
   resists chunking — bracket it with a sample before/after and accept one
   ~1,449 cy gap (it runs on ≤ one Jovian per frame). Total fix: ~4 emit-path
   polls × **103 cy measured** (`fc.py --cycles`, issue #508) ≈ **~412 cy/frame**
   (a fast-path `snd-poll` that finds no new HSYNC row returns in ~22 cy). Note
   this corrects the earlier optimistic ~17 cy/poll estimate — the real emit
   path is the full phase-advance + wavetable lookup + DAC write + NEXT.

4. **Piggyback the dense cluster on `wait-past-row`.** Space Warp already spins
   on the HSYNC flag once per frame (`spacewarp.fs:5159` →
   `kernel.asm:1757 WAIT-PAST-ROW`) to beam-chase before drawing. That spin
   loops `LDA $FF01 / BPL / LDA $FF00 / DECB` for ~70-190 rows doing nothing
   useful — the ideal venue for a tight `snd-fill` burst that writes one
   HSYNC-locked sample per row, near-free.

See `proposals/hsync_polling_budget_chart.html` (Sample-spacing eval view) for
the visual: the four piercing blocks and the single poll point each one needs.

### Space Warp scheduling — evaluation

Space Warp is already a capable **frame-level** scheduler: even/odd phase split
(collisions odd, gravity both), round-robin Jovian think-slots (skip=1, one
Jovian per even frame), MOD-4 gravity, MOD-8 BG slots, MOD-16 repair. That
staggering is exactly why the peak frame sits at ~13,091 cy instead of spiking
when everything coincides. The limitation sound exposes is that the schedule is
**frame-atomic** — within a frame every word runs to completion, no sub-frame
yielding. For video that is correct (everything is VSYNC-locked). Audio is the
first workload that cares about *intra-frame* regularity, so the sound engine
adds a **second, finer tier beneath** the existing frame tier: keep the
frame-level round-robin as-is; add `snd-poll` at phase boundaries (already
~400-1,400 cy apart) and split only the four over-line blocks.

### `async-sound.fs` — library word set

Pure library, no kernel dependency. State cells are DP-located (`$0050-$007F`,
free per the memory map) so CODE words reach them in 4 cy.

| Word | Stack | Kind | Role |
|------|-------|------|------|
| `snd-async-init` | `( -- )` | CODE | PIA mux → 6-bit DAC (reuse `lib/sound.fs` `snd-init` body) |
| `snd-note` | `( freq amp frames -- )` | `:` | Start a voice: `freq>inc` → `snd-inc`, set amp + frame count, reset phase, mark active. Returns immediately |
| `snd-stop` | `( -- )` | `:` | Silence: clear active, frames→0, DAC to mid-scale |
| `snd-frame` | `( -- )` | `:` | Per-VSYNC housekeeping: decrement `snd-frames`, auto-stop at 0, step envelope. Call once/frame |
| `snd-playing?` | `( -- f )` | `:` | Is a voice active (for game logic / gating) |
| `snd-poll` | `( -- )` | **CODE** | The workhorse. If a fresh HSYNC has passed, advance `snd-phase` by `snd-inc`, look up `snd-wave[phase_hi]`, shift by `snd-amp`, write `$FF20`. Sprinkled at phase boundaries; fast-path return when no new row |
| `snd-fill` | `( n -- )` | **CODE** | Tight loop: emit `n` HSYNC-locked samples back-to-back. For the `wait-past-row` spin / VSYNC-wait tail (the dense cluster) |
| `snd-pitch!` | `( freq -- )` | `:` | Live retune without restarting the note (continuous sweeps) |
| `snd-amp!` | `( amp -- )` | `:` | Live amplitude change (fades, accents) |
| `freq>inc` | `( freq -- inc )` | `:` | Hz → 16-bit phase increment for the chosen sample rate |
| `snd-wave` | `( -- addr )` | DATA | 256-byte wavetable (sine v1) via `DATA[PY`; square is special-cased to drop the table |

Internal DP cells: `snd-cur` (byte, current sample), `snd-phase` (16b),
`snd-inc` (16b, pitch), `snd-amp` (byte, shift), `snd-frames` (16b, remaining),
`snd-active` (flag), `snd-last-hs` (byte, last HSYNC flag state for edge detect).

The main loop touches only `snd-note` / `snd-stop` / `snd-frame` (events +
once-per-frame). `snd-poll` / `snd-fill` are the sub-frame placements the game
sprinkles at the four split points and into the `wait-past-row` spin. Multi-voice
(v2) clones the state block and sums candidates in `snd-poll` before the DAC write.

**Design decision — resolved as implemented (issue #508).** The shipped
`async-sound.fs` splits the two roles instead of forcing one word to do both:

- `snd-poll` is **HSYNC-edge-gated, non-blocking**: it emits one sample only if
  the `$FF01` flag shows a new scan line has elapsed (cleared via `$FF00`),
  otherwise returns in ~22 cy. It never blocks and never catches up more than
  one row, so it is safe to sprinkle anywhere; its effective sample rate is
  whatever the call cadence achieves.
- `snd-fill ( n -- )` is **fully HSYNC-locked, blocking**: it emits exactly `n`
  evenly-spaced samples, one per scan line, for the VSYNC-wait / `wait-past-row`
  spin where the CPU would idle anyway. This is the jitter-free path.

`freq>inc` converts Hz assuming the full 15,734 Hz (`snd-fill`) rate; sparser
`snd-poll` cadence lowers the effective rate and pitch proportionally, so a
voice driven mainly by `snd-poll` is calibrated per use site. This sidesteps the
"pitch drifts with frame content" problem by making the steady, even samples
come from `snd-fill` in the idle spin and treating `snd-poll` as opportunistic
fill-in.

## Why C, in detail

**Option C is the only path** to async continuous wavetable audio on
stock CoCo 1/2 hardware. The CoCo 3 has a programmable GIME timer
that can fire FIRQ at arbitrary rates (used by every CoCo 3 music
player and sample tracker found in the wild), but we have no
equivalent. Our only audio-rate interrupt source is HSYNC, fixed at
15,734 Hz.

Under R0=1 with 7,500 cy hard cap, two C variants both land near the
budget:

### C-(i): Synth in main loop, IRQ does zero-order-hold

The IRQ handler is **4 instructions**:

```
        LDA   <sample    ; 4 cy   DP-located sample byte (DP=$00)
        STA   $FF20      ; 5 cy   write DAC
        LDA   $FF00      ; 5 cy   read PIA0 PA → clears CA1 flag
        RTI              ; 6 cy
;  FIRQ entry (auto-push CC+PC): 7 cy
;  total: 27 cy per IRQ
```

Per frame: **27 cy × 262 IRQs = 7,074 cy.** That's the floor; nothing
can drop below it because the four operations (sample fetch, DAC
write, PIA ack, RTI) are individually mandatory.

Synthesis happens in **main-loop time**, outside the FIRQ. The main
loop updates `<sample>` at the effective audio rate (4 kHz = ~67
updates per frame, ~15 cy each = ~1,005 cy of main-loop work). The
IRQ unconditionally writes the same byte 4 times before main loop
updates it — natural zero-order-hold upsampling from 4 kHz to
15.7 kHz, exactly what a real DAC reconstruction filter expects.

| Cost component | cy/frame |
|---|---:|
| IRQ (262 × 27 cy) | 7,074 |
| Main-loop synth (67 × ~15 cy) | 1,005 |
| **Total** | **~8,080** |

### C-(ii): Synth in handler, run at 1/4 rate via skip-counter

Handler always pays a 27 cy fast-path; every 4th call does the
wavetable lookup + DAC write.

| Cost component | cy/frame |
|---|---:|
| 197 fast-path IRQs × 27 cy | 5,319 |
| 65 full-path IRQs × 57 cy | 3,705 |
| **Total** | **~9,024** |

### Active vs silent — the hidden lever

Both variants quote *active-frame* cost. When no voice is playing we
**disable PIA0 CA1 IRQ** (`AND <$FF01, #$FE`), bringing the per-frame
overhead to ~0 cy. In a typical space shooter, sound is active maybe
20–40% of the time (laser bursts, explosions, alerts), so the
amortized cost across a play session is much lower than the
active-frame peak.

| Scenario | Frame cy | % of frame |
|---|---:|---:|
| C-(i) active, sound playing | ~8,080 | 36% |
| C-(ii) active, sound playing | ~9,024 | 41% |
| Either, silent (IRQ disabled) | ~20 | <1% |
| Long-term average (~30% active) | ~2,500 | ~11% |

The 36–41% peak is the genuine cost of asynchronous wavetable audio
on this hardware. There is no architectural trick that makes it
cheaper without losing audio quality. Both variants are over the
nominal 7,500 cy budget by ~600–1,500 cy; the tradeoff is whether
that overrun lives in the game-logic column (synth in main loop,
variant i) or the sound column (synth in handler, variant ii).

**Recommendation: C-(i)** — the smaller IRQ handler, easier to
debug, and synthesis lives in normal Forth/CODE where state can be
inspected. ~600 cy of variant-(i)'s overrun is the main-loop synth,
which fits naturally inside the game-logic budget without changing
the IRQ side.

## API sketch

The user-facing model has four 16-bit state variables per voice
(8 bytes per voice, all DP-located for cheap access from main-loop
synth):

```forth
\ Per-voice state (one voice initially; multi-voice = clone this block)
VARIABLE snd-phase      \ phase accumulator (low byte = fractional)
VARIABLE snd-freq       \ phase increment per sample-update tick
VARIABLE snd-duration   \ remaining frames; 0 = idle, voice silent
VARIABLE snd-amplitude  \ 0..63, applied as a right-shift on the wavetable byte

\ Wavetable: 256 bytes, one period of a chosen waveform.
\ Indexed by high byte of snd-phase.
DATA[PY snd-wavetable
  ... 256 bytes of waveform ...
]DATA

\ Public API ─────────────────────────────────────────────────────────

\ Begin a voice playing.  Returns immediately.
: snd-play  ( freq duration amplitude -- )
  snd-amplitude !  snd-duration !  snd-freq !
  0 snd-phase !
  snd-enable ;

\ Main-loop tick — call ~67 times per frame (every ~330 main cycles)
\ to update the next-sample byte the FIRQ will pick up.
: snd-update  ( -- )
  snd-duration @ 0= IF EXIT THEN
  snd-phase @ snd-freq @ + DUP snd-phase !  \ advance phase
  8 RSHIFT snd-wavetable + C@               \ look up table[phase_hi]
  snd-amplitude @ RSHIFT                    \ scale by amplitude
  <sample C! ;                              \ stash for the FIRQ

\ Per-frame housekeeping — call once per VSYNC to decrement duration.
: snd-frame-tick  ( -- )
  snd-duration @ 0> IF
    snd-duration @ 1- snd-duration !
    snd-duration @ 0= IF snd-disable THEN
  THEN ;

\ Enable / disable the HSYNC FIRQ.
CODE snd-enable
        LDA   $FF01
        ORA   #$01
        STA   $FF01
        ANDCC #$BF              ; unmask FIRQ
        ;NEXT
;CODE

CODE snd-disable
        LDA   $FF01
        ANDA  #$FE
        STA   $FF01
        ;NEXT
;CODE
```

Multi-voice (v2) sums two `<sample>` candidates from two voices
before storing, doubling the synth cost (~2,000 cy) and the
amplitude headroom requirement.

## Cost analysis

Frame breakdown for Space Warp with the engine active:

| Component                          | Cycles | Frame % |
|------------------------------------|-------:|--------:|
| VSYNC wait + `snd-frame-tick`      |    ~80 |   0.4%  |
| FIRQ handler (262 × 27 cy)         |  7,074 |  31.9%  |
| Main-loop `snd-update` calls       | ~1,005 |   4.5%  |
| Game logic                         | 14,041 |  63.2%  |

The engine adds **~36 %** to every active frame and ~0% to silent
frames (CA1 IRQ disabled). The handler runs in RAM at 0.894 MHz; the
~7,100 cy figure already accounts for that.

## Kernel interaction

Implementing C requires sorting out a few things the current kernel
holds:

1. **FIRQ vector** — must point at our handler. On ROM mode the
   vector is at `$FFF6-$FFF7`. Either expose a kernel hook that
   installs the FIRQ handler address, or use the RAM trampoline at
   `$010C/$010D` (CoCo BASIC's user-FIRQ vector — already pointed at
   by ROM, free to redirect).

2. **`ORCC #$50` mask** — the kernel's inner loop runs with both IRQ
   and FIRQ masked. We need a careful unmask: clear F (FIRQ) but
   leave I (IRQ) alone, via `ANDCC #$BF`. Done once at `snd-enable`,
   re-asserted by `snd-disable`.

3. **Y register** — the kernel's ITC inner loop uses Y as scratch.
   FIRQ does *not* auto-save Y, so the handler **must not touch Y**.
   The 4-instruction handler above uses only A — safe. Any future
   in-handler synth (variant ii) must `PSHS Y / PULS Y` if it needs
   Y, adding 11 cy per IRQ → reconsider the budget.

4. **DP register** — kernel uses DP=$00. Our `<sample>` byte must
   live in $0000-$00FF (kernel has free DP slots at $0070-$007F per
   the memory map). PIA reads/writes in the handler are extended
   addressing, which is fine.

## Implementation plan

1. **Issue + branch.** File a single tracking issue ("Async sound
   engine: HSYNC FIRQ, C-(i) variant") that owns the work and
   references this document.

2. **Kernel-side prep.** Add a `kvar-firq` cell exposed via
   `kvar-*` so the library can swap the FIRQ handler in/out without
   touching ROM. Document the Y-register and DP constraints next to
   it.

3. **Library `lib/sound-async.fs`.** New file — keeps the existing
   synchronous lib untouched. Defines `snd-phase`, `snd-freq`,
   `snd-duration`, `snd-amplitude`, the wavetable (256-byte sine for
   v1), `snd-play`, `snd-update`, `snd-frame-tick`, `snd-enable`,
   `snd-disable`.

4. **Demo `src/sound-async/`.** A demo that runs a moving sprite +
   ongoing tone simultaneously, proving the engine doesn't block the
   game loop. Sweep across the wavetable to demonstrate continuous
   pitch control.

5. **Space Warp integration (optional v1.5).** Hook `snd-play` into
   thrust, fire, and impact events. Profile actual frame-time
   degradation on real hardware.

6. **Migration.** Existing demos that use synchronous `snd-beep` etc.
   keep working; the async lib is purely additive.

## Open questions

- **Multi-voice.** C-(i) extends to two voices by running
  `snd-update` twice (one per voice) and summing into `<sample>`.
  Cost roughly doubles in the main-loop column (~2,000 cy), IRQ
  stays at 7,074. Worth doing in a v2 once v1 lands.

- **Wavetable choice.** v1 ships a sine table (256 bytes, generated
  by `DATA[PY` at compile time). Alternates worth considering for v2:
  square (1 byte, special-cased — saves 256 bytes), pulse-width-
  variable square, saw, triangle, noise (LFSR in lieu of table).

- **Envelopes.** Per-frame `snd-frame-tick` already runs; an ADSR
  byte index added to it is ~10 cy/frame extra. Cheap.

- **Music engine on top.** Once `snd-play` exists with a real
  frequency parameter, a sequencer (read note from a buffer, set
  freq, set duration, advance) is straightforward — that's what
  issue #37 (PLAY music command) would be built on.

- **Polling fallback.** For demos that don't need continuous async
  audio, Option B (HSYNC polling at known yield points in the game
  loop) is cheaper at runtime and trivially compatible with the
  current kernel. Worth documenting separately as a low-overhead
  option for games structured around a tight main loop.
